## 用户输入

用户可以指定 issue ID，也可以从当前 git 分支自动推断。

## 概述

本 skill 通过交叉验证代码变更与需求、技术方案、测试方案和编码规范，进行全面的代码审核，生成详细的审核文档。

## 前置条件

- Pull Request 已创建
- `release-note.md` 存在（或至少代码变更已在分支上）

## 工作流程

### 第 1 步：定位 Issue 并收集材料

1. 读取 `.issuekit/config.yaml` 中的 `issues_dir` 配置项，获取 Issue 文档存放目录（默认为 `issues`）
2. 定位 issue 目录（从用户输入或当前分支，issue 目录位于 `{issues_dir}/{issue-id}/`）
3. 阅读所有 issue 文档：`requirement.md`、`technical-design.md`、`test-plan.md`、`release-note.md`
4. 获取代码差异：`git diff master...HEAD`
5. 获取 PR 信息（如有）：`gh pr view`

### 第 2 步：七维度审核

#### 2.1 需求吻合度
- 逐条对照用户故事和验收标准检查代码变更
- 验证每个 User Story 的流程是否已实现
- 标记未实现或部分实现的需求

#### 2.2 方案一致性
- 对照技术方案验证实际实现
- 检查接口设计是否与实现一致
- 检查数据库变更是否与设计一致
- 标记偏离

#### 2.3 测试覆盖度
- 验证测试方案是否覆盖所有需求点
- 检查关键代码路径是否有对应测试用例
- 识别未覆盖区域

#### 2.4 代码质量
- 命名规范（类名/方法名/变量名）
- 异常处理（不吞异常、有意义的错误信息）
- 日志记录（关键路径有日志、级别合理）
- 边界条件（空值检查、数组越界、并发安全）
- 代码重复（可提取的公共方法）

#### 2.5 安全审查
- SQL 参数化（无字符串拼接 SQL）
- XSS 防护（输出转义）
- 敏感数据（密码/token 不明文、不日志）
- 权限校验（接口鉴权完整）

#### 2.6 性能审查
- 无 N+1 查询问题
- 大批量操作有分批处理
- 热点数据有缓存策略
- 数据库索引覆盖查询条件

#### 2.7 兼容性审查
- API 向后兼容
- 数据库迁移安全
- 客户端版本兼容

### 第 3 步：生成审核文档

加载模板 `.issuekit/templates/code-review.md` 并填入审核结果。

### 第 4 步：写入文件并报告

1. 将 `code-review.md` 写入 issue 目录
2. 报告审核结论：
   - **通过**：所有维度通过
   - **有条件通过**：有必须修复项，修复后通过
   - **不通过**：发现严重问题
